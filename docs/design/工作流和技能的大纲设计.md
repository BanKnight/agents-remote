# 工作流和技能的大纲设计

## 1) 阶段链路

Intent（原始意图） → Roadmap（版本与变更编排） → Step（读取 context/progress 并分发阶段技能） → Specify（what） → Design（how） → Build（实现） → Verify（一致性） → Distill（长期沉淀） → Archive（版本归档）

`docs/project.md` 是 project big picture，可在任意阶段通过 `describe-project` 渐进式补全；参与需求讨论、roadmap、spec、design、implementation、verify 或 distill 前，应先读取它建立项目整体认知；change 完成后也应按需提炼更新。

### 关键原则

- `.workflow/intents.md` 只保存尚未进入 roadmap 的原始意图，分为“待讨论”和“待分配”。
- `plan-versions` 必须全量加载 `.workflow/intents.md`；不要只从底部往上读或只处理最近意图，避免遗漏意图关系、重复规划或错误分配。
- `.workflow/versions/index.md` 是 workflow big picture，是当前活跃 versions/changes 的索引和当前焦点入口，不是历史总账，也不维护单个 change 的阶段状态。
- `progress.md` 是单个 change 的阶段状态、局部阻塞、产物检查和进展记录的权威来源；不保存下一步技能。
- `context.md` 是单个 change 的看板上下文，说明 change 为什么存在、承接哪些来源、已知边界、协作共享方式和背景引用；不写成 spec、design、plan 或任务清单。
- `step-change` 是推荐的 change 推进入口：它读取 `.workflow/versions/index.md` 当前焦点或用户指定 change 的 `context.md` / `progress.md`，再调用对应阶段技能。
- Agent 不能只机械消费 intents；当 big picture 不足以有效规划时，需要主动规划调研、设计、验证、CI/CD、文档治理或其他补齐 big picture 的 changes。
- Roadmap 层与 change 层本质上都是“获取意图 → 规划 → 实现/验证 → 沉淀”的循环；roadmap 从全局意图池和 big picture 缺口出发，change 从 `context.md` 这张看板卡出发。
- 一个 change 进入实现前，应先经过 `specify-change` 明确 what，再按需经过 `design-change` 明确 how。
- 实现后先 Verify，再 Distill。
- 归档以 version 为单位，归档后从活跃 versions 移入 `.workflow/archive/versions/`；旧归档结构如已存在，保持不改。

## 2) 已实现 Skills

| Skill | 主要职责 | 输入 | 输出 | 退出条件 |
|---|---|---|---|---|
| help-workflow | 介绍整套工作流与可用技能，并根据用户描述分流到合适技能 | 用户问题/当前描述 | 工作流说明与推荐命令 | 用户理解当前阶段和可用技能；不读取项目状态 |
| setup-workflow | 初始化/更新 `.workflow/` 与 `docs/` 两个关键目录，并注入 `CLAUDE.md` / `AGENTS.md` 加载规则 | 项目现状 + `.claude/workflow/templates/workflow/` + `.claude/workflow/templates/docs/` | `.workflow/` + `docs/` 基础结构 + 入口加载区块 | 两区结构、治理文件、模板、索引与入口加载规则可用 |
| describe-project | 交互式建立或更新项目认知 big picture | 用户项目认知 + `docs/templates/project.md` + 现有 `docs/project.md` | `docs/project.md` | 项目定位、用户场景、领域概念或长期准则得到渐进式补全 |
| clarify-intents | 记录尚未进入 roadmap 的原始意图 | 口述需求/问题 + `.workflow/intents.md` + `.workflow/templates/intents.md` + 按条件加载 `docs/project.md`、docs 索引链路、运行态上下文或外部资料 | `.workflow/intents.md` | 意图进入“待讨论”或“待分配”，或建议进入 `plan-versions` |
| plan-versions | 从全局 big picture 视角全量读取 intents，编排活跃 versions/changes，并建立或更新 change 看板上下文和进度骨架 | `.workflow/intents.md` + `.workflow/versions/index.md` + versions/change 模板 + `.workflow/versions/` + `docs/project.md` + 按需 docs/archive/code/外部资料 | `.workflow/versions/index.md` + `.workflow/versions/<version>/changes/<change-id>/context.md` + `.workflow/versions/<version>/changes/<change-id>/progress.md` + 按需 `.workflow/versions/<version>/shared/` | 活跃 index 与 version/change 目录一致；已处理意图进入 roadmap、保留待分配、暂缓或放弃且原因明确 |
| step-change | 读取当前或指定 change 的 context/progress，判断阶段并分发到对应阶段技能 | `.workflow/versions/index.md` + `.workflow/versions/<version>/changes/<change-id>/context.md` + `.workflow/versions/<version>/changes/<change-id>/progress.md` + change artifacts | 调用对应阶段技能 + 更新 `.workflow/versions/<version>/changes/<change-id>/progress.md` | 当前 change 推进一阶段、阻塞明确或需要用户选择 |
| specify-change | 产出单个 change 的行为契约（what） | `.workflow/versions/index.md` + `.workflow/versions/<version>/changes/<change-id>/context.md` + spec 模板 | `.workflow/versions/<version>/changes/<change-id>/specs/<capability>/spec.md` | 每条需求可验证且有场景 |
| design-change | 产出单个 change 的 how（架构/接口/数据/交互方案） | change context + specs + version shared + 项目现状 + design 子域模板 | `.workflow/versions/<version>/changes/<change-id>/design/<subdomain>.md` | 需要的设计子域已创建或更新，关键决策、权衡、风险齐全 |
| plan-change | 为单个 change 产出实现计划并拆解任务，将 specs/design/docs/version shared 的上游承诺投影到可执行任务 | change context + specs + design + version shared + 相关 `docs/` + plan/tasks 模板 | `.workflow/versions/<version>/changes/<change-id>/plan.md` + `.workflow/versions/<version>/changes/<change-id>/tasks.md` | plan 提供足够实现上下文，tasks 可独立完成并验收，且包含任务承诺清单 |
| implement-change | 按单个 change 的 plan/tasks 实施代码变更，按任务类型组合加载 references，执行前提取并完成前核对任务承诺清单 | `.workflow/versions/<version>/changes/<change-id>/context.md` + `plan.md` + `tasks.md` + plan 中列出的必要上下文 + 适用 implementation references | 代码变更 + tasks 勾选 | 所有任务完成或阻塞明确，任务承诺已核对，可进入 verify-change |
| verify-change | 通过 harness 与证据验证 change 实现是否符合 spec/design/tasks | 代码 + change context + specs + design + plan/tasks + 验证 harness | `.workflow/versions/<version>/changes/<change-id>/verify.md` + 按需 artifacts | 无 CRITICAL 问题，或问题已分级并明确回流 |
| distill-change | 提炼单个 change 的长期知识 | change context + specs + design + verify 证据 + 实现结果 + version shared + 相关 `docs/` | `docs/specs/*` + `docs/design/*` + `docs/architecture/*` + `docs/runbooks/*` + 按需 `docs/project.md` | 可长期复用的 WHAT / HOW / project knowledge 已沉淀 |
| archive-version | 以 version 为单位冻结上下文并迁入 archive | 完成的 version + version shared + 全部关联 changes | `.workflow/archive/versions/<version>/` | version 已归档且活跃 index 已移除 |

## 4) 关键文件职责

| 文件 / 目录 | 职责 |
|---|---|
| `docs/project.md` | project big picture，记录项目定位、用户场景、领域概念与长期准则，渐进式补全。 |
| `.workflow/intents.md` | 只保存尚未进入 roadmap 的原始意图，分为“待讨论”和“待分配”；`plan-versions` 必须全量加载。 |
| `.workflow/versions/index.md` | 当前活跃 versions/changes 的索引，标记当前焦点、活跃 versions、全局阻塞、暂缓/放弃和下一步入口；不维护单个 change 阶段状态或完整上下文。 |
| `.workflow/versions/<version>/shared/` | version 内多个 changes 共享的运行态材料；producer/consumer 关系由各 change 的 `context.md` 显式约定。 |
| `.workflow/versions/<version>/changes/<change-id>/context.md` | 单个 change 的看板上下文：来源上下文、当前已知边界、协作与共享上下文、背景引用；不写 spec/design/plan/tasks。 |
| `.workflow/versions/<version>/changes/<change-id>/progress.md` | 单个 change 的阶段状态、局部阻塞、产物检查和进展记录，是 `step-change` 的决策入口。 |
| `.workflow/versions/<version>/changes/<change-id>/specs/` | 本 change 的行为契约增量。 |
| `.workflow/versions/<version>/changes/<change-id>/design/` | 本 change 的 how 设计材料，按子域拆分。 |
| `.workflow/versions/<version>/changes/<change-id>/plan.md` | 本 change 的实现计划，必须列出实现所需上下文、version shared 和已沉淀 docs 引用。 |
| `.workflow/versions/<version>/changes/<change-id>/tasks.md` | 本 change 的可执行任务清单，由 plan-change 生成。 |
| `.workflow/versions/<version>/changes/<change-id>/verify.md` | 本 change 的一致性证据。 |
| `.workflow/versions/<version>/changes/<change-id>/artifacts/` | 本 change 的截图、基准、测试报告、trace 等证据材料。 |
| `.workflow/archive/versions/<version>/` | 已归档 version 的完整运行态上下文，包含 shared 与 changes。 |
| `docs/` | 长期沉淀区，保存主线规格、长期设计、架构、ADR、runbooks 等。 |
| `.claude/skills/<skill-name>/` | workflow 技能包，包含 `SKILL.md` 以及按需 bundled `references/`、`templates/`。 |
| `.claude/skills/setup-workflow/templates/workflow/` / `.claude/skills/setup-workflow/templates/docs/` | setup-workflow 使用的模板源。 |

## 6) Intent / Roadmap / Version / Change 关系

```text
.workflow/intents.md
  └─ 只保存待讨论 / 待分配原始意图，plan-versions 全量加载
       ↓ plan-versions
.workflow/versions/index.md
  └─ 活跃 versions/changes 索引、当前焦点、依赖、全局阻塞、暂缓/放弃
       ↓ step-change 读取当前焦点或指定 change 的 progress.md
.workflow/versions/<version>/changes/<change-id>/context.md
  └─ 本 change 的看板上下文：来源、边界、协作共享、引用入口
.workflow/versions/<version>/changes/<change-id>/progress.md
  └─ 本 change 的阶段状态、局部阻塞、产物检查和进展记录
       ↓ step-change 调用对应阶段技能
.workflow/versions/<version>/shared/
  └─ 同 version 内 producer/consumer changes 共享的运行态材料
```

规则：

- intent 一旦被 `plan-versions` 编排进 roadmap、暂缓或放弃，就从 `.workflow/intents.md` 移除；如果需要等待前置 change 补齐 big picture，仍可保留在 `.workflow/intents.md` 并说明阻塞。
- `plan-versions` 必须全量理解 `.workflow/intents.md` 后再判断哪些进入版本、哪些保留待分配、哪些暂缓或放弃。
- `.workflow/versions/index.md` 中每个活跃 change 必须对应 `.workflow/versions/<version>/changes/<change-id>/`。
- `.workflow/versions/index.md` 只维护队列、依赖、当前焦点、全局阻塞、暂缓/放弃；change 完整上下文由 `context.md` 管理，阶段状态由 `progress.md` 管理。
- `step-change` 读取 `.workflow/versions/index.md` 当前焦点或用户指定 change 的 `progress.md`，再调用 `specify-change` / `design-change` / `plan-change` / `implement-change` / `verify-change` / `distill-change` 等阶段技能推进一步。
- change 不一定直接来自 intent；也可以来自 Agent 主动规划的技术铺垫、调研、设计、验证、CI/CD、文档治理、迁移准备、工程整理或 big picture 补齐工作。
- 当某个 change 需要向同 version 其他 changes 提供共享材料时，在 `context.md` 的“需要写入 shared”中说明路径、内容和使用方；消费方在“需要读取 shared”中说明路径和用途。
- big picture 由 Agent 按需读取 `docs/`、`.workflow/versions/index.md`、version shared、change context/progress、archive 和项目文件得到，而不是靠 `intents.md` 保存全部历史。

## 7) Version / Change 关系

- version 是当前开发队列中的可收口开发批次。
- 一个 version 可以包含多个 changes，且可以有 version 内共享材料。
- 归档以 version 为单位进行，目标路径是 `.workflow/archive/versions/<version>/`。
- change 是单一可验证变更单元。
- change-id 是语义化变更标识，不是数字编号；即使路径位于 version 下，活跃区仍保持全局唯一。
- change 从 `context.md` 开始执行；`context.md` 像看板系统里的单条卡片，给承接该 change 的 AI 提供必要上下文。
- 非用户意图直接来源的 change 必须在 `context.md` 的“主动规划上下文”中说明背景、需要解决的问题和支撑的后续目标。
- 同 version 内 change 之间需要通信时，通过 `.workflow/versions/<version>/shared/` 共享材料，并由各自 `context.md` 显式约定读写。

## 8) 运行态 vs 长期沉淀

| 类型 | 位置 | 生命周期 | 更新方式 |
|---|---|---|---|
| project big picture | `docs/project.md` | 长期演进 | 由 describe-project 渐进式补全，change 完成后按需提炼 |
| 待处理意图 | `.workflow/intents.md` | 进入 roadmap 前；若需等待前置 change 补齐 big picture，可继续保留 | 由 clarify-intents 写入，由 plan-versions 全量读取并按处理结果移出或保留 |
| 活跃路线图 | `.workflow/versions/index.md` | version 开发期间 | 由 plan-versions 维护 |
| version 共享运行态材料 | `.workflow/versions/<version>/shared/` | version 未归档前 | 由 producer change 写入，consumer change / roadmap 按 context 读取；长期价值经 distill 进入 docs |
| 活跃 change | `.workflow/versions/<version>/changes/<change-id>/...` | change 未归档前 | 由后续 workflow 命令逐步补齐 |
| change 看板上下文 | `.workflow/versions/<version>/changes/<change-id>/context.md` | change 未归档前 | 由 plan-versions 创建/更新；后续阶段可在不改变职责边界的前提下补充引用 |
| 长期知识 | `docs/specs/...` + `docs/design/...` + `docs/architecture/...` + `docs/runbooks/...` + 按需 `docs/project.md` | 持续演进 | 由 distill-change 更新 |
| 历史追溯 | `.workflow/archive/versions/<version>/...` | 永久保留 | 由 archive-version 生成；旧 archive 结构如已存在，保持不改 |
