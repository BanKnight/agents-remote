# 工作流和技能的大纲设计

## 1) 阶段链路

Intent（原始意图） → Roadmap（版本与变更编排） → Step（读取 progress 并分发阶段技能） → Specify（what） → Design（how） → Build（实现） → Verify（一致性） → Distill（长期沉淀） → Archive（版本归档）

`docs/project.md` 是 project big picture，可在任意阶段通过 `describe-project` 渐进式补全；参与需求讨论、roadmap、spec、design、implementation、verify 或 distill 前，应先读取它建立项目整体认知；change 完成后也应按需提炼更新。

### 关键原则

- `intents.md` 只保存尚未进入 roadmap 的原始意图。
- `docs/project.md` 是 project big picture，参与需求讨论、roadmap、spec、design、implementation、verify 或 distill 前应先读取。
- `roadmap.md` 是 workflow big picture，是当前活跃 versions/changes 的索引和当前焦点入口，不是历史总账，也不维护单个 change 的阶段状态。
- `progress.md` 是单个 change 的阶段状态、下一步技能和局部阻塞的权威来源。
- `step-change` 是推荐的 change 推进入口：它读取 roadmap 当前焦点或指定 change 的 `progress.md`，再调用对应阶段技能。
- Agent 不能只机械消费 intents；需要基于项目现状主动规划必要的铺垫、验证、质量和治理 changes。
- 一个 change 进入实现前，应先经过 `specify-change` 明确 what，再按需经过 `design-change` 明确 how。
- 实现后先 Verify，再 Distill。
- 归档以 version 为单位，归档后从活跃 roadmap 移入 archive。

## 2) 已实现 Skills

| Skill | 主要职责 | 输入 | 输出 | 退出条件 |
|---|---|---|---|---|
| help-workflow | 介绍整套工作流与可用技能，并根据用户描述分流到合适技能 | 用户问题/当前描述 | 工作流说明与推荐命令 | 用户理解当前阶段和可用技能；不读取项目状态 |
| setup-workflow | 初始化/更新 `.workflow/` 与 `docs/` 两个关键目录，并注入 `CLAUDE.md` / `AGENTS.md` 加载规则 | 项目现状 + `.claude/workflow/templates/workflow/` + `.claude/workflow/templates/docs/` | `.workflow/` + `docs/` 基础结构 + 入口加载区块 | 两区结构、治理文件、模板、索引与入口加载规则可用 |
| describe-project | 交互式建立或更新项目认知 big picture | 用户项目认知 + `docs/templates/project.md` + 现有 `docs/project.md` | `docs/project.md` | 项目定位、用户场景、领域概念或长期准则得到渐进式补全 |
| clarify-intents | 记录尚未进入 roadmap 的原始意图 | 口述需求/问题 + `.workflow/templates/intents.md` | `.workflow/intents.md` | 意图进入“待讨论”或“待分配” |
| plan-roadmap | 编排活跃 versions/changes，并建立或更新 change 骨架 | `.workflow/intents.md` + `docs/` + `.workflow/roadmap.md` + `.workflow/changes/` + 按需 archive | `.workflow/roadmap.md` + `.workflow/changes/<change-id>/intents.md` + `.workflow/changes/<change-id>/progress.md` | 活跃 roadmap 与 changes 目录一致 |
| step-change | 读取当前或指定 change 的 progress，判断阶段并分发到对应阶段技能 | `.workflow/roadmap.md` + `.workflow/changes/<change-id>/progress.md` + change artifacts | 调用对应阶段技能 + 更新 `.workflow/changes/<change-id>/progress.md` | 当前 change 推进一阶段、阻塞明确或需要用户选择 |
| specify-change | 产出单个 change 的行为契约（what） | `.workflow/roadmap.md` + `.workflow/changes/<change-id>/intents.md` + spec 模板 | `.workflow/changes/<change-id>/specs/<capability>/spec.md` | 每条需求可验证且有场景 |
| design-change | 产出单个 change 的 how（架构/接口/数据/交互方案） | change intents + specs + 项目现状 + design 子域模板 | `.workflow/changes/<change-id>/design/<subdomain>.md` | 需要的设计子域已创建或更新，关键决策、权衡、风险齐全 |
| plan-change | 为单个 change 产出实现计划并拆解任务 | change specs + change design + 相关 `docs/` + plan/tasks 模板 | `.workflow/changes/<change-id>/plan.md` + `.workflow/changes/<change-id>/tasks.md` | plan 提供足够实现上下文，tasks 可独立完成并验收 |
| implement-change | 按单个 change 的 plan/tasks 实施代码变更 | `.workflow/changes/<change-id>/plan.md` + `.workflow/changes/<change-id>/tasks.md` + plan 中列出的必要上下文 | 代码变更 + tasks 勾选 | 所有任务完成或阻塞明确，可进入 verify-change |
| verify-change | 通过 harness 与证据验证 change 实现是否符合 spec/design/tasks | 代码 + change specs + design + plan/tasks + 验证 harness | `.workflow/changes/<change-id>/verify.md` | 无 CRITICAL 问题，或问题已分级并明确回流 |
| distill-change | 提炼单个 change 的长期知识 | change specs + change design + verify 证据 + 实现结果 + 相关 `docs/` | `docs/specs/*` + `docs/design/*` + `docs/architecture/*` + 按需 `docs/project.md` | 可长期复用的 WHAT / HOW / project knowledge 已沉淀 |
| archive-version | 以 version 为单位冻结上下文并迁入 archive | 完成的 version + 全部关联 changes | `.workflow/archive/roadmap.md` + `.workflow/archive/changes/` | version 已归档且活跃 roadmap 已移除 |

## 4) 关键文件职责

| 文件 / 目录 | 职责 |
|---|---|
| `docs/project.md` | project big picture，记录项目定位、用户场景、领域概念与长期准则，渐进式补全。 |
| `.workflow/intents.md` | 只保存尚未进入 roadmap 的原始意图，分为“待讨论”和“待分配”。 |
| `.workflow/roadmap.md` | 当前活跃 versions/changes 的索引，标记当前焦点、活跃 versions、全局阻塞和下一步入口；不维护单个 change 阶段状态。 |
| `.workflow/changes/<change-id>/intents.md` | 单个 change 的来源记录：可能来自用户意图，也可能来自 Agent 主动规划出的铺垫、验证、质量或治理工作。 |
| `.workflow/changes/<change-id>/progress.md` | 单个 change 的阶段状态、下一步技能、局部阻塞和进展记录，是 `step-change` 的决策入口。 |
| `.workflow/changes/<change-id>/specs/` | 本 change 的行为契约增量。 |
| `.workflow/changes/<change-id>/design/` | 本 change 的 how 设计材料，按子域拆分。 |
| `.workflow/changes/<change-id>/plan.md` | 本 change 的实现计划，必须列出实现所需上下文和已沉淀 docs 引用。 |
| `.workflow/changes/<change-id>/tasks.md` | 本 change 的可执行任务清单，由 plan-change 生成。 |
| `.workflow/archive/roadmap.md` | 已归档 versions 的索引。 |
| `.workflow/archive/changes/` | 已归档 changes 的完整上下文。 |
| `docs/` | 长期沉淀区，保存主线规格、长期设计、架构、ADR、runbooks 等。 |
| `.claude/skills/<skill-name>/` | workflow 技能包，包含 `SKILL.md` 以及按需 bundled `references/`、`templates/`。 |
| `.claude/skills/setup-workflow/templates/workflow/` / `.claude/skills/setup-workflow/templates/docs/` | setup-workflow 使用的模板源。 |

## 6) Intent / Roadmap / Change 关系

```text
.workflow/intents.md
  └─ 只保存待讨论 / 待分配意图
       ↓ plan-roadmap
.workflow/roadmap.md
  └─ 活跃 versions/changes 索引与当前焦点
       ↓ step-change 读取 progress.md
.workflow/changes/<change-id>/progress.md
  └─ 本 change 的当前阶段、下一步技能与局部阻塞
       ↓ step-change 调用对应阶段技能
.workflow/changes/<change-id>/intents.md
  └─ 本 change 的来源意图或规划来源
```

规则：

- intent 一旦被 `plan-roadmap` 编排进 roadmap，就从 `.workflow/intents.md` 移除。
- roadmap 中每个活跃 change 必须对应 `.workflow/changes/<change-id>/`。
- roadmap 只维护队列、依赖和当前焦点；change 阶段状态和下一步技能由 `changes/<change-id>/progress.md` 管理。
- `step-change` 读取 roadmap 当前焦点或用户指定 change 的 `progress.md`，再调用 `specify-change` / `design-change` / `plan-change` / `implement-change` / `verify-change` / `distill-change` 等阶段技能推进一步。
- change 不一定直接来自 intent；也可以来自 Agent 主动规划的铺垫、验证、质量或治理工作。
- big picture 由 Agent 按需读取 `docs/`、`.workflow/roadmap.md`、`.workflow/changes/`、`.workflow/archive/` 得到，而不是靠 `intents.md` 保存全部历史。

## 7) Version / Change 关系

- version 是当前开发队列中的可收口开发批次。
- 一个 version 可以包含多个 changes。
- 归档以 version 为单位进行。
- change 是单一可验证变更单元。
- change-id 是语义化变更标识，不是数字编号。
- 非 intent 直接来源的 change 必须在 `changes/<change-id>/intents.md` 中写明规划来源。

## 8) 运行态 vs 长期沉淀

| 类型 | 位置 | 生命周期 | 更新方式 |
|---|---|---|---|
| project big picture | `docs/project.md` | 长期演进 | 由 describe-project 渐进式补全，change 完成后按需提炼 |
| 待处理意图 | `.workflow/intents.md` | 进入 roadmap 前 | 由 clarify-intents 写入，由 plan-roadmap 移出 |
| 活跃路线图 | `.workflow/roadmap.md` | version 开发期间 | 由 plan-roadmap 维护 |
| 活跃 change | `.workflow/changes/<change-id>/...` | change 未归档前 | 由后续 workflow 命令逐步补齐 |
| 长期知识 | `docs/specs/...` + `docs/design/...` + `docs/architecture/...` + 按需 `docs/project.md` | 持续演进 | 由 distill-change 更新 |
| 历史追溯 | `.workflow/archive/roadmap.md` + `.workflow/archive/changes/...` | 永久保留 | 由 archive-version 生成 |
