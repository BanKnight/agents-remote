# UI/UX Design

## Change

- change-id：align-mobile-app-shell

## 页面 / 界面范围

- 登录后首页 / Project 列表入口。
- 全局登录后 shell 的移动端布局约束。
- 与后续 Project 控制台、Session detail、Files/Git 页面共享的移动端密度和溢出基线。

## 页面结构

- 移动端 shell 使用全高深色 App 容器：页面背景、主体内容和导航/操作区域应像一个控制台工作台，而不是普通网页文档流。
- 首页结构按优先级组织：
  1. 顶部紧凑上下文区：产品/账户/当前入口的最小识别信息。
  2. 主内容区：已有 Project 或工作上下文列表。
  3. 次级操作区：Create/Adopt Project 入口，采用轻量按钮、折叠面板或小卡片。
- 移动端内容区域应允许纵向滚动，但页面不应出现横向滚动。
- 固定底部导航或操作区需要预留内容底部 padding，避免遮挡最后一项内容。

## 交互模式

- Project 列表项是首页主点击目标，触控区域应足够大，并清晰表达 Project 名称与可进入状态。
- Create/Adopt Project 入口默认不展开为大表单；用户点击后再显示完整输入区域或进入对应流程。
- 页面切换和状态反馈优先使用清晰文本、标签和按钮状态，不只依赖颜色。
- 原型中的视觉层级可转换为本项目界面：顶部上下文、主列表/会话卡片、状态标签、底部入口，但文案和对象必须替换为 Project/Agent/Terminal/Files/Git。

## 页面状态

- 默认态：已有 Project 时，首页首屏优先展示 Project 列表和紧凑上下文；低频创建/采用入口处于次级但可发现位置。
- 加载态：Project 数据加载时保留 shell 结构，主内容区展示列表骨架或紧凑加载提示，避免整页大空白跳动。
- 空态：无 Project 时，展示简短说明和创建/采用入口；空态可以比已有项目态更突出创建/采用，但仍保持移动端密度。
- 错误态：Project 加载或创建失败时展示可读错误和重试/返回路径；错误提示不应撑开横向布局。
- 成功态：创建或采用 Project 成功后，用户能看到新 Project 或进入该 Project，不停留在占用首屏的大表单。

## 可用性要求

- 手机竖屏下不出现页面级横向滚动；长名称、路径或状态文本应截断、换行或放入局部滚动容器。
- 顶部上下文区和底部区域不得共同挤压主内容到不可用高度；主内容至少保留可操作列表区域。
- 触控目标应适合手指点击，尤其是 Project item、Create/Adopt 入口和主要导航入口。
- 状态标签必须带文字，例如 `Agent Sessions`、`Terminal`、`Files`、`Git`、`Create Project`，避免只用图标。
- 宽屏布局可以扩展为居中最大宽、侧栏或双栏，但不得破坏移动端定义的语义顺序。

## 关键决策

- 将首页从“页头 + 大创建/采用区域 + Project 列表”改为“紧凑上下文 + Project 主列表 + 次级创建/采用入口”。
- 不把原型图的具体内容逐项复刻；只迁移暗色、紧凑、App-like、状态可见的信息层级。
- 全局 shell 的布局基线优先解决溢出和可达性，再考虑更精细视觉 polish。

## 风险与权衡

- 如果创建/采用入口过度隐藏，会影响首次使用；设计要求它次级但可发现，不允许完全消失。
- 如果 App-like shell 固定区域过多，会压缩主内容；因此固定区域必须克制，长内容交给主区域滚动。
- 不在本 change 细化 Files/Git/Session 详情的全部视觉规则，避免与后续专门 change 重叠。

## 开放问题

- 无。

## 后续沉淀候选

- 移动端 App-like shell 页面结构和状态规则。
- 移动端页面级横向不溢出、固定区不遮挡内容的长期 UI/UX 准则。
