## 组件边界

- shadcn/ui components should be treated as local source components generated by CLI, not as a black-box theme to inherit wholesale.
- Project console primitives wrap or compose shadcn/Radix behaviors when accessibility or interaction semantics matter.
- Reusable components are justified only when they serve cross-page prototype fidelity or reduce real duplication.
- Page-specific layout and copy stay close to the page to preserve prototype nuance.
- Business/API/query code must not be pulled into generic UI primitives.

## 状态管理

- Keep current server state in TanStack Query.
- Keep URL-visible Project workspace state in TanStack Router route/search.
- Keep single-page UI details such as input drawer collapsed, selected file preview, or local overlay state in component state unless current docs already require URL persistence.
- Keep Jotai limited to shell-level shared UI state.
- Do not introduce new state libraries for this version.

## 路由 / 页面接入

- Do not proactively change routes or search state.
- Make minimal route/search adjustments only if required to express prototype navigation/return semantics.
- Direct Project workspaces and deep/contextual details must remain mutually clear in mobile navigation.

## 工程约束

- `implement-change` for React/prototype alignment must load `vercel-react-best-practices` before coding or reviewing.
- shadcn/ui setup should use the official CLI path for Vite/Tailwind v4 and create `components.json` if absent.
- Context7 shadcn docs confirm Vite setup includes `@` alias in `vite.config.ts` and global CSS with Tailwind v4 theme variables; current project does not yet have `@` alias or `components.json`.
- Context7 lucide docs confirm named icon imports from `lucide-react` are ESM tree-shakable and icons accept SVG props such as `size`, `color`, `strokeWidth`, and `absoluteStrokeWidth`.
- npm metadata checked 2026-05-28 showed `shadcn@4.8.2` published 2026-05-27 and `lucide-react@1.17.0` published 2026-05-28, both inside the 7-day safety window. Implementation planning must re-check versions and either select an older safe version where feasible or wait/confirm before using a too-new latest.
- shadcn-related dependencies likely include CLI-generated component dependencies such as Radix packages, `class-variance-authority`, `clsx`, `tailwind-merge`, and `tw-animate-css`; exact dependencies must be determined by the minimal component set at plan/implementation time.
- Existing Tailwind entry is `web/src/styles/index.css`; design-system CSS variables should integrate there or in a clearly imported adjacent CSS file rather than scattering tokens across pages.

## 关键决策

- Do not initialize a complete design system upfront; create a thin baseline and let page changes correct it through shared updates.
- Use shadcn/ui for semantics and accessibility, but make project-specific wrappers/tokens responsible for the console visual language.
- Use lucide-react through a local icon primitive so future replacement or tuning is centralized.
- Keep dependency introduction deferred to implementation plan so supply-chain checks can use current package metadata.

## 风险与权衡

- shadcn defaults can pull the UI toward a generic dashboard; wrappers and tokens must keep the dark Server Agent Console tone.
- Too much abstraction can flatten prototype-specific nuance; the non-abstraction list is part of the design contract.
- Avoiding route/state changes reduces risk, but page changes must still be allowed to make minimal fixes if prototype navigation cannot otherwise work.
- Latest npm package versions may be too new under the 7-day rule; implementation may need explicit version choices.

## 开放问题

- The exact shadcn component set should be selected during page-level planning after reading current UI code.
- Whether `@` alias should be introduced only for shadcn compatibility or broader web imports should be decided in implementation planning.
- Existing lint/format setup for generated shadcn components needs confirmation before implementation.
