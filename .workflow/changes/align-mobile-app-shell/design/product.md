# Product Design

## Change

- change-id：align-mobile-app-shell

## 用户目标 / 产品目标

- 用户在手机上打开控制台时，应像打开一个远程 Agent 控制 App，而不是进入一个桌面网站缩放版。
- 用户最常见目标是快速进入已有 Project 并继续观察或控制 Agent/Terminal 相关工作；创建或采用 Project 是低频任务，不应占据移动端首屏主导位置。
- 本 change 为后续 Project 工作区、Session 控制台和 Files/Git 移动优化建立统一产品层级。

## 功能边界

### 做什么

- 修改登录后首页的信息层级：Project/工作上下文优先，Create/Adopt Project 次级但可发现。
- 明确 App-like shell 的产品语义：页面围绕 Project、Agent Sessions、Terminal、Files、Git 组织。
- 建立原型术语映射规则：原型只提供移动端气质和层级参考，不提供最终产品文案。
- 定义移动端页面不溢出的产品体验基线：用户不应因为布局撑开或固定区遮挡而无法访问核心内容。

### 不做什么

- 不改变 Project 创建或采用的能力范围、校验规则或服务端行为。
- 不新增 Project 最近访问、收藏、排序、搜索或分组能力。
- 不新增 Agent/Terminal 运行态能力或 provider history/resume。
- 不定义 Files/Git 只读详情的信息密度细节。
- 不引入用户自定义导航、快捷键配置或主题配置。

## 用户流程

- 默认路径：用户登录后进入首页，优先浏览已有 Project 或当前工作上下文，点击目标 Project 后进入 Project 控制台。
- 低频路径：用户需要创建或采用 Project 时，从首页次级入口进入创建/采用流程，完成输入后回到 Project 列表或进入对应 Project。
- 视觉参考路径：实现者参考原型时，先识别原型中的结构意图，再把页面概念替换为本项目术语。
- 回退路径：如果 Project 列表加载失败或为空，首页仍提供可理解的错误/空态和创建/采用入口。

## 信息架构

- 顶层首页表达“这是远程 Agent 控制台的 Project 入口”，而不是营销型 landing page。
- Project 是首页主对象；创建/采用是围绕 Project 的次级行动。
- Project 内能力保持长期语义：Agent Sessions 是默认主焦点，Terminal、Files、Git 是同一 Project 作用域下的辅助入口。
- Prototype 中任何不属于本项目领域的词、卡片或示例内容都不得成为最终信息架构节点。

## 体验路径

- 首次或空项目用户：看到明确空态和创建/采用入口，但空态不应变成大面积营销页。
- 已有项目用户：首屏优先看到可进入的 Project 列表或最近上下文，低频创建/采用入口以按钮、折叠区、轻量卡片或等价方式存在。
- 移动用户：可单手识别当前位置、核心对象和下一步入口，不需要横向滚动或缩放页面。
- 宽屏用户：保留同样语义，可展示更多辅助信息，但不改变主路径。

## 关键决策

- Create/Adopt Project 在移动首页降级为次级入口，而不是删除；这样保留新用户路径，同时减少已有用户日常操作负担。
- 首页页头从品牌/说明导向改为工作上下文导向；用户最关心的是进入 Project，而不是阅读产品介绍。
- 原型参考的优先级低于本项目术语、安全边界和长期 specs；任何冲突都按本项目文档处理。

## 风险与权衡

- 降低创建/采用入口视觉占比可能让新用户首次发现成本略升；通过空态、次级按钮和清晰标签缓解。
- App-like shell 容易诱导加入过多导航 chrome；本 change 只建立基线，不提前承诺完整 native app 动效或离线能力。
- 为所有页面建立溢出基线可能暴露既有页面布局问题；本 change 只修全局和首页，后续 change 继续收敛具体页面。

## 开放问题

- 无。

## 后续沉淀候选

- 移动端首页主路径与低频 Project 创建/采用入口的长期产品原则。
- 原型参考必须映射到项目领域术语的产品规则。
