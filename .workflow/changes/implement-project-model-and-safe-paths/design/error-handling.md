# Error Handling Design

## Change

- change-id：implement-project-model-and-safe-paths

## 异常范围

本设计覆盖 Project 列表、创建/采用、详情解析和 project-safe-path 解析中的失败行为。

不覆盖：登录失败、WebSocket 认证失败、Files/Git/Session 子资源失败；这些由对应能力设计处理，但必须复用本 change 的路径边界错误语义。

## 失败场景

### 用户输入错误

- 请求 body 不是 JSON 或缺少 `path`。
- `path` 为空字符串。
- Project 名称包含路径分隔语义，导致它不是一级目录名。
- 用户输入绝对路径但目标不是 `PROJECTS_ROOT` 的一级子目录。

处理：返回 `400`，提示用户输入 `PROJECTS_ROOT` 下的一级文件夹名称或一级子目录路径。

### 资源错误

- 详情接口请求的 Project 不存在。
- 目标已存在但不是目录。
- 列表枚举时某个条目在读取过程中被外部删除。

处理：详情不存在返回 `404`；创建目标为文件返回 `400`；列表枚举遇到瞬时变化时可以跳过消失条目或返回文件系统错误，但不得把非目录当成 Project。

### 边界错误

- 输入路径越出 `PROJECTS_ROOT`。
- Project 名称经解析后越出 root。
- Project-relative path 经解析后越出 Project 根目录。
- 符号链接或真实路径导致最终目标越界。

处理：返回 `400` 或 `403` 均可，计划阶段应统一选择；错误语义必须明确为 path boundary violation，并且停止后续文件系统操作。

### 并发冲突

- 创建前目标不存在，创建时另一个进程创建了同名目录。
- 创建前目标不存在，创建时出现同名文件。
- 检查时是目录，返回前目录被删除。

处理：如果最终目标是目录，可按采用成功返回；如果最终目标不是目录，返回冲突或目标无效错误。

### 系统/文件系统错误

- `PROJECTS_ROOT` 不存在、不可读或权限不足。
- 创建目录失败。
- 真实路径解析失败。
- 文件系统返回未知错误。

处理：对用户返回通用可行动消息；日志保留 error code、operation、projectName 或相对路径等上下文。

## 错误码 / 错误语义

建议扩展：

| 错误码 | 类型 | HTTP | 可重试 | 调用方动作 |
|---|---|---:|---|---|
| `PROJECT_NAME_INVALID` | 用户输入错误 | 400 | 否 | 修改 Project 名称或路径 |
| `PROJECT_TARGET_INVALID` | 用户输入/资源错误 | 400 | 否 | 选择一级目录，避免文件/root/嵌套目录 |
| `PROJECT_PATH_OUTSIDE_ROOT` | 边界错误 | 400/403 | 否 | 选择 `PROJECTS_ROOT` 内一级目录 |
| `PROJECT_NOT_FOUND` | 资源错误 | 404 | 否 | 返回列表或创建目录 |
| `PROJECT_CONFLICT` | 并发冲突 | 409 | 可在用户确认后重试 | 重新加载目标状态 |
| `PROJECT_FS_ERROR` | 文件系统错误 | 500 | 视原因而定 | 稍后重试或检查服务器权限 |

错误 response 沿用现有 `{ error: { code, message } }` 形态，不在本 change 引入复杂 error details。

## 重试 / 降级 / 恢复

- Project 列表失败不使用缓存降级；目录状态必须反映当前服务器文件系统，失败应明确提示。
- 创建目录对用户重复点击应尽量幂等：如果第二次请求看到目录已存在且是目录，返回成功。
- 越界、嵌套、目标为文件等用户输入错误不可自动重试。
- 文件系统瞬时错误可以由用户手动重试，不设计后台重试任务。
- 不做补偿/回滚：本 change 的唯一副作用是创建目录；目录创建成功后不因后续摘要构造失败而删除目录。

## 用户可见反馈

- 未认证：回到登录页或提示需要重新登录，沿用 auth 能力。
- Project 不存在：提示 Project 不存在，可返回 Project 列表或重新创建。
- 目标越界：提示只能选择 `PROJECTS_ROOT` 下一级目录。
- 目标为文件：提示该名称已被文件占用，需要换一个文件夹名。
- 文件系统权限失败：提示服务器无法读取或创建项目目录，需要检查服务运行用户和目录权限。

## 关键决策

- 路径边界错误必须在执行后续 project-scoped 操作前失败，不能由调用方自行判断。
- 重复创建已存在目录按成功处理，降低移动端重复点击或网络重试带来的误报。
- 不在错误响应中返回堆栈、内部模块名或无关外部路径。
- 对已认证用户，成功响应可以返回 Project 真实路径；失败响应只返回必要的可行动信息。

## 风险与权衡

- 如果只做字符串前缀校验，符号链接可能造成越界；计划阶段必须覆盖真实路径解析测试。
- 过度隐藏错误会降低个人部署排错效率；折中方案是 response 给可行动摘要，服务端日志保留详细错误上下文。
- 并发文件系统变化无法完全避免；设计上以最终状态为准，而不是依赖检查时快照。

## 开放问题

- 计划阶段需统一 `PROJECT_PATH_OUTSIDE_ROOT` 使用 `400` 还是 `403`；建议若错误来自用户输入则用 `400`，若未来来自权限边界则可用 `403`。
