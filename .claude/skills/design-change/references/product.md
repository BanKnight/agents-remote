# 产品设计

本文件用于指导产品设计。当设计任务涉及用户目标、功能边界、产品流程、信息架构、体验路径时加载。

## 适用场景

- 澄清一个产品要做什么、不做什么。
- 定义用户是谁、解决什么问题、为什么现在要做。
- 划分新增能力和修改已有能力的边界。
- 把产品目标展开为用户故事和可验证需求。

## 输入检查

进入产品设计前，先确认以下输入是否可用：

- 用户或 clarify 阶段提供的原始意图。
- `.workflow/decisions.md` 中相关的原始决策。
- 现有 `docs/` 中相关的产品文档、调研、ADR。
- 项目已有的业务术语和概念定义。

如果输入不够，先建议回到 clarify，不要自行补全用户意图。

## 产品设计关注点

### 用户是谁

明确目标用户群体，不要只写“所有用户”。

区分不同角色的目标和行为差异，例如：

- 管理员 vs 普通用户。
- 付费用户 vs 免费用户。
- 内部运营 vs 外部客户。

### 要解决什么问题

从用户视角描述问题，不是从系统视角。

不好的写法：“需要实现用户管理模块。”
好的写法：“管理员无法批量禁用过期账号，每次只能逐个操作，当有大量离职员工时非常耗时。”

### 为什么现在要做

说明时机和驱动力，例如：

- 用户反馈或投诉集中在这个痛点。
- 竞品已有类似功能。
- 合规或安全要求。
- 依赖这个能力的后续功能被阻塞。

### 新增和修改哪些能力

把产品能力分成两组：

- **新增能力**：之前不存在，需要从零构建的。
- **修改能力**：已有功能的行为发生变化，需要说明变了什么。

每个能力用一句话描述覆盖范围。

### 用户路径

描述用户完成某个目标的关键路径。

这里不需要写完整的界面和交互细节，但要覆盖：

- 起点和终点。
- 路径中的关键步骤和分支。
- 哪些步骤涉及系统自动行为（不需要用户操作）。
- 哪些步骤可能失败或需要回退。

### 什么不做

显式写出非目标，这对后续设计和版本规划都很重要。

非目标的来源通常是：

- 本次不做的已知需求。
- 被讨论过但被否决的方向。
- 属于其他能力范围的功能。

### 可验证需求

产品设计中最重要的质量要求：需求必须可验证。

每个关键需求至少包含一个场景，场景用 WHEN/THEN 表达：

```markdown
### Requirement: 管理员可以批量禁用用户

#### Scenario: 批量禁用选中的过期用户

- WHEN 管理员在用户列表中选中多个状态为“已过期”的用户，点击“批量禁用”
- THEN 系统将选中用户的状态改为“已禁用”，并记录操作日志
```

```markdown
#### Scenario: 不允许批量禁用活跃用户

- WHEN 管理员在用户列表中选中包含活跃用户的多个用户，点击“批量禁用”
- THEN 系统拒绝操作，提示“只能禁用过期或已停用的用户”
```

## 工作方式

### 保留用户原意

产品设计文档的核心是用户的意图和问题，不是 Agent 的解读。

如果用户说“我想要一个批量操作的功能”，不要把它改写成“实现基于 CRUD 的批量资源管理模块”。保持用户的语言风格，只在结构化和精确表达上做润色。

### 使用已有业务术语

如果项目中已经有业务术语定义，在产品设计中沿用这些术语。

如果发现术语不一致或含糊，指出冲突，不要静默选择一种用法。

### 遇到模糊词要追问

当用户使用“比较好”“一般来说”“通常”这类词时，追问具体含义。

追问的方式：

- “你说‘比较好的用户体验’，具体是指什么场景下、什么操作应该更快？”
- “你提到‘通常情况’，那非通常情况是什么？”

### 用具体场景压测

当产品边界不清晰时，构造具体的极端场景来测试边界：

- “如果用户同时属于两个角色，这个功能应该按哪个角色的权限执行？”
- “如果批量操作中途失败了，已经处理的部分是回滚还是保留？”

### 不写实现细节

产品设计中不写技术实现、代码结构、API 路径、数据库表名。

例外：当某个实现约束会影响产品行为时（例如“搜索结果可能有 500ms 延迟”），可以说明这个产品体验约束，但不展开实现方案。

## 补充检查点

本 reference 不规定输出结构；输出结构由 `design-change` 命令本体与 `.workflow/templates/changes/design/` 统一决定。

使用产品视角时，重点确认：

- 是否保留用户原始意图。
- 是否明确用户、问题和做这件事的原因。
- 是否区分新增能力和修改能力。
- 是否描述关键用户路径。
- 是否明确非目标。
- 关键需求是否能被后续 spec 或 verify 验证。
