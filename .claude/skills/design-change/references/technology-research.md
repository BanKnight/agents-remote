# 技术版本与依赖安全检索规则

`design-change` 不能仅依赖模型记忆做技术选型、版本选择或依赖引入。涉及框架、库、运行时、工具链、外部服务或 npm 包时，必须先检索当前资料并核对项目现状。

## 触发条件

出现以下情况时必须执行检索：

- 新增依赖、替换依赖或升级依赖。
- 选择框架、库、SDK、运行时或构建工具版本。
- 使用模型可能过期的 API、配置或最佳实践。
- specs/design 要求接入外部服务、浏览器 API、数据库、消息队列、认证方案等。
- 涉及 npm、PyPI、Cargo、Go modules 等包生态的安全风险。

## 检索内容

至少确认：

- 当前稳定版本。
- 官方推荐安装和配置方式。
- 当前版本的 breaking changes、迁移要求和弃用项。
- 与项目现有技术栈的兼容性。
- 包发布时间与近期安全事件。
- 是否有更成熟、维护更活跃或更少依赖的替代方案。

## npm 依赖安全规则

由于 npm 包投毒、账户劫持、供应链攻击等风险，新增或升级 npm 依赖时必须额外检查：

- 目标版本发布时间。
- 版本发布距今是否至少 7 天。
- 最近版本是否存在异常高频发布、维护者变更、包名相似混淆或安全公告。
- 是否可以使用项目已有依赖替代。
- 是否可以避免新增依赖，用平台能力或少量项目内代码解决。

默认规则：

- 不选择发布不足 7 天的 npm 版本，除非用户明确确认。
- 不为了轻微便利新增依赖。
- 不使用不明来源、维护不活跃或下载/使用信号明显不足的包。

## 资料来源优先级

优先使用：

1. 官方文档与 release notes。
2. 包管理器元数据（版本、发布时间、维护者、依赖）。
3. 官方 GitHub 仓库、security advisory、issue / discussion。
4. 可信生态资料或社区迁移指南。

不要把博客、旧问答或模型记忆作为唯一依据。

## 补充检查点

本 reference 不规定输出结构；输出结构由 `design-change` 命令本体与 `.workflow/templates/changes/design/` 统一决定。

涉及技术版本或依赖时，重点确认：

- 是否检索官方文档、release notes、包元数据或安全公告。
- 是否记录候选方案、选定版本或版本范围。
- 是否说明选择原因和不选其他常见方案的原因。
- 是否判断安全与供应链风险。
- npm 依赖是否满足发布至少 7 天规则。
- 如果版本不足 7 天但仍选择，是否记录用户确认和原因。
