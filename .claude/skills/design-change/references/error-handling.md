# 异常处理设计

本文件用于指导异常处理设计。当设计任务涉及异常场景、错误码、失败处理、重试、降级、边界情况时加载。

## 适用场景

- 需要定义用户操作失败时的系统行为。
- 需要设计 API 错误码、错误消息或错误恢复方式。
- 需要处理外部服务失败、网络超时、并发冲突、数据不一致等问题。
- 需要定义重试、降级、补偿、回滚或告警策略。
- 需要明确哪些错误对用户可见，哪些只用于日志和监控。
- 需要设计批量操作、异步任务或多步骤流程的失败处理。

## 输入检查

进入异常处理设计前，先确认以下输入是否可用：

- 产品设计中的用户路径和失败场景。
- 业务规则中的禁止路径、状态约束和冲突规则。
- API 设计中的接口、调用方和错误格式。
- 架构设计中的模块边界、外部依赖和同步/异步流程。
- 数据库设计中的事务边界、唯一约束和迁移风险。
- 项目已有错误码、日志、监控、告警和重试约定。

如果只知道成功路径，不要直接补异常设计；先补齐失败场景。

## 异常处理关注点

### 错误分类

先分类错误，再设计处理方式。

常见分类：

- **用户输入错误**：参数缺失、格式错误、非法值。
- **权限错误**：未登录、无权限、资源不属于当前用户。
- **业务规则错误**：状态不允许、额度不足、规则冲突。
- **资源错误**：资源不存在、已删除、重复创建。
- **并发冲突**：版本冲突、重复提交、状态已变化。
- **外部依赖错误**：第三方服务失败、超时、限流。
- **系统内部错误**：未知异常、数据损坏、配置错误。

不同错误类型应有不同处理策略。不要所有错误都返回“系统错误”。

### 用户可见行为

异常处理不只是技术行为，也影响产品体验。

每个用户可见错误要说明：

- 用户看到什么提示。
- 用户是否能理解原因。
- 用户下一步可以做什么。
- 是否需要保留用户已输入内容。
- 是否允许重试。
- 是否需要联系客服或管理员。

错误提示应该可行动，不要只写“操作失败”。

### 调用方可处理行为

API 或服务间错误要让调用方能处理。

每个错误应说明：

- 错误码。
- 错误类型。
- 是否可重试。
- 是否幂等安全。
- 调用方建议动作。
- 是否应该记录、告警或展示给最终用户。

### 重试策略

不是所有失败都应该重试。

重试设计要说明：

- 哪些错误可以重试。
- 哪些错误不能重试。
- 最大重试次数。
- 退避策略。
- 是否需要抖动。
- 重试是否可能造成重复副作用。
- 重试失败后的最终状态。

### 降级策略

当依赖不可用时，说明系统是否可以降级。

降级策略要明确：

- 降级后保留哪些能力。
- 暂停哪些能力。
- 用户如何感知。
- 是否需要展示缓存数据。
- 是否会影响数据正确性。
- 如何恢复正常。

不要为了“高可用”随意降级。如果降级会产生错误业务结果，应失败而不是降级。

### 补偿和回滚

多步骤流程失败时，要说明如何处理已完成步骤。

常见策略：

- 全部回滚。
- 保留已成功部分，标记部分失败。
- 发起补偿动作。
- 进入人工处理队列。
- 记录待修复状态，由后台任务继续处理。

批量操作尤其需要说明：全部成功、部分成功、全部失败分别如何表达。

### 幂等和重复提交

异常处理必须考虑重复请求。

需要说明：

- 用户重复点击时会发生什么。
- 网络超时后再次提交是否安全。
- 异步任务重复投递是否安全。
- 第三方回调重复发送是否安全。
- 幂等键或业务唯一键是什么。

### 可观测性

异常设计要覆盖如何发现和定位问题。

说明：

- 哪些错误需要日志。
- 哪些错误需要告警。
- 日志中必须包含哪些上下文。
- 是否需要 traceId/requestId。
- 哪些错误只计数监控，不需要打扰值班人员。
- 哪些错误需要人工介入。

### 安全与隐私

错误信息不能泄露敏感信息。

避免：

- 暴露数据库结构。
- 暴露内部服务名或堆栈。
- 暴露用户是否存在于系统中（在敏感场景）。
- 暴露权限判断细节。
- 在日志中记录密钥、token、身份证号、完整银行卡等敏感数据。

## 工作方式

### 先列失败场景

不要从错误码开始设计。

优先顺序：

1. 用户路径或系统流程。
2. 每一步可能失败的原因。
3. 失败后用户或调用方应该怎么办。
4. 系统是否重试、降级、补偿或告警。
5. 最后再定义错误码和格式。

### 区分可恢复和不可恢复

每个错误都要判断是否可恢复。

- 可恢复错误：用户改输入、重新登录、稍后重试、等待异步处理。
- 不可恢复错误：业务状态不允许、资源不存在、权限永久不足。
- 需要人工介入：数据不一致、外部系统长期失败、资金或合规相关异常。

### 不吞掉错误

降级、重试和兜底不能让错误消失。

即使用户看到的是友好提示，系统内部也应该保留足够信息用于排查。

### 不为不可能场景设计

只处理系统边界、外部依赖、用户输入、并发和数据一致性等真实风险。

不要给内部不可能状态写大量防御性分支，除非它来自真实历史问题或外部输入。

## 补充检查点

本 reference 不规定输出结构；输出结构由 `design-change` 命令本体与 `.workflow/templates/changes/design/` 统一决定。

使用异常处理视角时，重点确认：

- 失败场景、影响对象和可恢复性是否明确。
- 用户可见错误、API/服务错误和内部日志是否区分。
- 重试、降级、补偿、回滚和幂等是否有边界。
- 可观测性、告警和人工介入条件是否明确。
- 错误提示和日志是否避免泄露安全或隐私信息。
- 是否避免为不可能场景设计大量防御性分支。

## 质量检查

完成异常处理设计后，检查：

- 是否先列失败场景，再定义错误码。
- 是否区分用户输入、权限、业务规则、外部依赖、并发和内部错误。
- 是否说明用户可见提示和下一步动作。
- 是否说明调用方如何处理错误。
- 是否定义重试、降级、补偿、回滚和幂等策略。
- 是否覆盖批量、异步、多步骤流程的失败处理。
- 是否包含日志、监控、告警和人工介入条件。
- 是否避免泄露敏感信息。
- 是否避免为不可能场景过度设计。
