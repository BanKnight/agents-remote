# UI/UX 设计

本文件用于指导 UI/UX 设计。当设计任务涉及页面体验、交互路径、信息层级、视觉规范、可用性、响应式体验或用户操作反馈时加载。

## 适用场景

- 把产品设计中的用户路径转化为页面和交互方案。
- 定义页面结构、信息层级、导航方式和关键操作入口。
- 设计表单、列表、详情页、仪表盘、设置页等界面体验。
- 约束视觉表现：颜色、字体、间距、圆角、阴影、组件状态等。
- 检查用户体验是否清晰、一致、可理解、可操作。
- 用户提供原型图、截图、草图或页面设计规范，需要将其转成可沉淀的设计文档。

## 输入检查

进入 UI/UX 设计前，先确认以下输入是否可用：

- 产品设计中的目标用户、用户路径、关键需求和非目标。
- 用户提供的页面草图、原型图、截图、设计稿或品牌规范。
- 现有 `docs/` 中的 UI/UX、品牌、组件或设计系统文档。
- 项目现有页面、组件库、前端框架和样式约束。

如果缺少产品目标或用户路径，不要直接做视觉设计；先补齐用户要完成什么任务。

## UI/UX 设计关注点

### 用户任务

先明确用户在这个页面或流程中要完成什么任务。

不要从“页面上应该有什么组件”开始，而是从“用户为什么来到这里、要做成什么事”开始。

需要说明：

- 用户进入页面的入口。
- 用户的主要目标。
- 用户完成目标所需的最短路径。
- 用户可能中断、返回、取消或失败的节点。

### 信息层级

界面应该让用户一眼知道：

1. 当前在哪里。
2. 现在最重要的信息是什么。
3. 可以做什么。
4. 下一步应该点哪里。

设计时区分：

- 主信息：页面存在的核心原因。
- 辅助信息：帮助用户判断或确认的上下文。
- 操作信息：按钮、链接、表单、筛选、批量操作等。
- 反馈信息：成功、失败、加载中、空状态、禁用状态等。

### 页面结构

根据页面目标选择合适结构，不要默认堆卡片。

常见结构包括：

- 任务流：适合多步骤操作、提交、审核、配置。
- 列表 + 详情：适合管理、检索、选择对象。
- 仪表盘：适合概览、监控、对比关键指标。
- 表单页：适合输入、编辑、配置。
- 向导页：适合复杂任务分步完成。
- 空状态引导：适合新用户首次进入或暂无数据。

### 交互状态

每个关键交互都应考虑状态，而不是只设计理想态。

至少考虑：

- 默认态。
- 悬停态。
- 聚焦态。
- 激活态。
- 加载态。
- 成功态。
- 失败态。
- 空状态。
- 禁用态。
- 权限不足态。

### 反馈与可恢复性

用户操作后应得到明确反馈。

重点说明：

- 操作是否成功。
- 失败时用户知道为什么失败。
- 用户是否可以撤销、重试或返回。
- 危险操作是否需要确认。
- 批量操作是否需要展示部分成功、部分失败。

### 可访问性

UI/UX 设计不只服务视觉正常、鼠标操作的用户。

至少考虑：

- 文本和背景对比度。
- 键盘可操作性。
- 表单 label 和错误提示。
- 焦点顺序。
- 不只依赖颜色表达状态。
- 移动端触控目标大小。

### 响应式体验

设计时说明不同屏幕下的结构变化。

至少考虑：

- 桌面端。
- 平板或窄屏。
- 移动端。

不要只说“响应式适配”，要说明关键区域如何变化：导航是否折叠、表格是否改为卡片、侧栏是否变成抽屉、批量操作入口是否保留。

## 视觉规范关注点

如果项目已有视觉规范，优先沿用。

如果需要定义新的页面视觉规范，至少覆盖：

### 视觉风格

说明整体气质，例如：

- 专业、克制、可信。
- 轻量、友好、低压。
- 高密度、效率优先。
- 强视觉、营销导向。

避免只写“现代化”“高级感”这类无法执行的词。每个风格词都应该能落到具体视觉规则。

### 颜色

说明颜色角色，而不是只列色值。

至少区分：

- 主色：用于主要行动、当前选中、关键高亮。
- 辅助色：用于次级行动、辅助信息。
- 中性色：用于背景、边框、分隔、文本层级。
- 语义色：成功、警告、错误、信息。

### 字体与文字层级

说明标题、正文、辅助文字、标签、按钮文字的层级。

重点不是列很多字号，而是保证阅读顺序稳定：

- 页面标题。
- 区块标题。
- 列表项标题。
- 正文。
- 描述/元信息。
- 标签/徽标。

### 布局与间距

说明布局模型和间距节奏。

例如：

- 页面最大宽度。
- 内容区与导航区关系。
- 区块之间的垂直间距。
- 卡片或表单内部 padding。
- 列表项密度。

### 形状与层级

说明圆角、边框、阴影、分层方式。

不要默认大量使用阴影；如果使用阴影，要说明它表达什么层级。如果不用阴影，也要说明用什么替代，例如边框、背景层级或间距。

### 组件状态

关键组件需要说明状态变化：

- Button。
- Input。
- Select。
- Tabs。
- Table。
- Card。
- Modal / Drawer。
- Toast / Alert。

不要只描述静态样式，要说明 hover、focus、disabled、loading、error 等状态。

## 原型图输入处理

当用户提供原型图、截图或设计稿时：

1. 先描述你观察到的页面结构和用户路径。
2. 区分“图中明确存在的内容”和“你推测的内容”。
3. 不要把图上的偶然布局当成规范，除非用户确认。
4. 如果图中存在歧义，提出具体问题。
5. 将确认后的内容沉淀为可复用的 UI/UX 设计说明。

## 工作方式

### 不用视觉替代产品判断

好看的页面不一定解决用户问题。

UI/UX 设计必须回到用户任务、信息层级和操作反馈上判断，而不是只优化颜色、圆角和排版。

### 先结构，后视觉

优先确定：

1. 页面承担什么任务。
2. 页面有哪些信息区域。
3. 用户的主路径是什么。
4. 哪些状态必须覆盖。

之后再定义视觉风格、颜色、字体和组件状态。

### 提供具体设计理由

每个关键 UI 决策都应说明原因。

不好的写法：“这里使用卡片布局，更美观。”
好的写法：“这里使用列表 + 右侧详情，因为用户需要快速比较多条记录，并在不离开列表的情况下查看详情。”

### 避免 AI 味视觉套路

不要默认使用：

- 大面积紫色渐变。
- 过度居中布局。
- 无意义的玻璃拟态。
- 所有元素统一大圆角。
- 过度空旷但信息密度不足的卡片墙。

除非这些确实符合产品气质和用户任务。

## 补充检查点

本 reference 不规定输出结构；输出结构由 `design-change` 命令本体与 `.workflow/templates/changes/design/` 统一决定。

使用 UI/UX 视角时，重点确认：

- 页面或流程服务的用户任务是否明确。
- 信息架构、页面结构和用户路径是否清楚。
- 关键交互是否覆盖默认、加载、成功、失败和禁用状态。
- 视觉规范是否服务于产品气质和用户任务。
- 响应式和可访问性是否有必要约束。
- 是否留下仍需产品、前端或实现确认的问题。
