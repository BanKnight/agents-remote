# 数据库设计

本文件用于指导数据库设计。当设计任务涉及数据模型、表结构、字段、关系、索引、迁移时加载。

## 适用场景

- 需要设计新的数据模型、表、集合或持久化结构。
- 需要修改已有数据库结构。
- 需要定义实体关系、字段语义、唯一约束或生命周期。
- 需要设计索引、查询模式、分页或性能优化。
- 需要规划数据迁移、回填、兼容和回滚。
- 需要判断业务规则应该由数据库约束、应用逻辑还是两者共同保证。

## 输入检查

进入数据库设计前，先确认以下输入是否可用：

- 产品设计中的能力、用户路径和可验证需求。
- 业务规则中的核心概念、状态和约束。
- 架构设计中的模块边界和数据归属。
- API 设计中的请求、响应和查询模式。
- 旧项目已有 schema、迁移工具、命名规范和数据访问方式。
- 数据量级、增长速度、查询频率、保留期限和合规要求。

如果核心实体、关系或查询模式不清楚，不要直接建表。

## 数据库设计关注点

### 概念模型先于表结构

先定义业务实体和关系，再映射为表或集合。

概念模型要回答：

- 有哪些核心实体。
- 每个实体代表什么业务概念。
- 实体之间是一对一、一对多还是多对多。
- 哪些实体有生命周期或状态。
- 哪些概念只是派生结果，不应该直接持久化。

### 数据归属

明确每类数据属于哪个模块、服务或业务上下文。

不要让多个模块随意读写同一张表。如果确实需要共享，说明：

- 谁拥有写权限。
- 谁只能读取。
- 通过什么接口或视图共享。
- 如何避免跨模块耦合。

### 表和字段语义

字段设计要表达业务含义，不只是类型。

每个重要字段应说明：

- 业务含义。
- 类型。
- 是否必填。
- 默认值。
- 是否可变。
- 是否可为空以及为什么。
- 有哪些合法取值。

不要使用含糊字段，例如 `status`、`type`、`metadata`，却不定义枚举含义或结构约束。

### 约束和不变量

数据库设计要说明哪些规则由数据库保证，哪些由应用层保证。

常见约束包括：

- 主键。
- 外键。
- 唯一约束。
- 非空约束。
- 检查约束。
- 软删除约束。
- 租户隔离约束。

如果某个业务不变量非常关键，应优先考虑是否能由数据库约束表达。

### 索引和查询模式

索引设计必须来自查询模式，不要凭感觉加索引。

每个索引要说明：

- 支持哪个查询。
- 过滤条件是什么。
- 排序条件是什么。
- 预期数据量级。
- 是否唯一。
- 写入成本是否可接受。

不要为了“可能以后会查”提前创建大量索引。

### 数据生命周期

说明数据如何创建、更新、归档和删除。

至少考虑：

- 数据什么时候创建。
- 哪些字段可以更新。
- 是否需要软删除。
- 是否需要审计记录。
- 是否需要历史版本。
- 数据保留多久。
- 是否涉及隐私、合规或删除请求。

### 迁移策略

旧项目数据库设计必须包含迁移策略。

说明：

- 是否新增表或字段。
- 是否修改已有字段语义。
- 是否需要回填历史数据。
- 是否需要兼容旧代码。
- 是否需要分阶段发布。
- 如何回滚。
- 迁移期间如何处理并发写入。

### 一致性和事务

说明哪些操作必须原子完成，哪些可以最终一致。

至少明确：

- 哪些写入必须在同一事务中完成。
- 哪些操作可以异步处理。
- 失败后是否需要补偿。
- 是否会出现重复事件或重复写入。
- 如何保证幂等。

## 工作方式

### 不从 UI 或 API 字段反推数据库

UI/API 字段不等于数据库字段。

先理解业务概念，再设计持久化模型。不要因为前端需要一个字段，就直接在表里加一个同名字段。

### 避免万能表和万能 JSON

`metadata`、`extra`、`payload` 这类字段可以用于扩展，但不能替代核心模型设计。

如果字段会被查询、约束、排序或参与业务规则，就不应该只藏在 JSON 里。

### 明确派生数据

如果某个字段可以由其他数据计算得出，要说明为什么需要存储。

存储派生数据通常需要说明：

- 性能原因。
- 历史快照原因。
- 审计原因。
- 外部系统同步原因。
- 如何保持一致。

### 设计可演进结构

数据库结构要避免过早复杂，也要避免明显堵死后续演进。

对未来扩展，只记录已经有现实依据的方向，不要为假设需求提前建复杂模型。

## 补充检查点

本 reference 不规定输出结构；输出结构由 `design-change` 命令本体与 `.workflow/templates/changes/design/` 统一决定。

使用数据视角时，重点确认：

- 是否先定义概念模型，再定义存储结构。
- 数据归属、字段业务含义和关键不变量是否明确。
- 查询模式、索引、数据生命周期是否有依据。
- 事务、一致性、幂等和并发写入是否被考虑。
- 旧项目是否说明迁移、回填、兼容和回滚。
- 是否避免万能 JSON 和过度设计。

## 质量检查

完成数据库设计后，检查：

- 是否先定义概念模型，再定义表结构。
- 是否明确数据归属。
- 是否定义字段业务含义，而不只是类型。
- 是否说明关键约束和不变量。
- 是否根据查询模式设计索引。
- 是否说明数据生命周期。
- 是否考虑事务、一致性和幂等。
- 是否为旧项目提供迁移和回滚策略。
- 是否避免万能 JSON 和过度设计。
