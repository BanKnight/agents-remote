# 前端设计

本文件用于指导前端设计。当设计任务涉及前端模块划分、组件边界、状态管理、路由、数据获取、渲染策略或前端工程约束时加载。

## 适用场景

- 将产品能力转化为前端页面、模块和组件结构。
- 设计前端状态管理、数据流和用户操作反馈。
- 设计页面路由、布局层级和前端导航结构。
- 明确前端与后端 API、缓存、权限、错误处理的协作方式。
- 处理已有前端代码的结构调整或复杂交互设计。

## 输入检查

进入前端设计前，先确认以下输入是否可用：

- 用户或 clarify 阶段提供的原始意图。
- 已有产品设计或用户路径。
- 现有前端项目结构、框架和约定。
- 已有页面、组件、路由、状态管理方式。
- 相关 API、数据模型或权限约束。

如果缺少产品目标或关键用户路径，先补齐上游设计，不要直接从组件开始设计。

## 技术栈识别

前端设计必须先识别技术栈，并在输出中显式记录。

至少确认：

- 前端框架：React、Vue、Svelte、Angular、Next.js、Nuxt、Remix 等。
- 语言与构建：TypeScript、JavaScript、Vite、Webpack、Turbopack 等。
- 路由方案：文件路由、声明式路由、服务端路由、客户端路由。
- 渲染模式：CSR、SSR、SSG、ISR、RSC、SPA、MPA。
- 状态管理：框架内置状态、Context、Redux、Zustand、Pinia、Signals 等。
- 数据请求与缓存：fetch、Axios、TanStack Query、SWR、GraphQL Client 等。
- 表单与校验：React Hook Form、Formik、Zod、Yup、Valibot 等。
- 样式系统：CSS Modules、Tailwind CSS、Sass、CSS-in-JS、组件库主题系统等。
- 组件库：shadcn/ui、MUI、Ant Design、Radix UI、Headless UI、自研组件库等。
- 测试工具：Vitest、Jest、Testing Library、Playwright、Cypress 等。

### 新项目

如果是新项目或用户要求选择技术栈，必须查询当前官方文档或可信来源，确认技术和版本没有过时。

设计时要说明：

- 推荐技术栈。
- 推荐原因。
- 不选择其他常见方案的原因。
- 当前版本或文档依据。
- 未来升级风险。

不要凭记忆推荐过时版本、过时脚手架或已经不推荐的实践。

### 旧项目

如果是旧项目，先以现有技术栈为约束进行查漏补缺。

检查重点包括：

- 当前技术栈是否仍能支撑需求。
- 关键依赖是否明显过时或已停止维护。
- 路由、状态、数据请求、样式、测试是否存在缺口。
- 是否有多个方案混用导致维护困难。
- 是否存在可以小步修复的问题，而不是推翻重写。

旧项目设计优先补齐缺口和对齐约定，不要轻易建议替换框架或大规模迁移。只有现有技术栈已经无法满足需求，才提出迁移方案，并明确迁移成本和风险。

## 前端设计关注点

### 页面与模块边界

先确定页面和模块，再确定组件。

需要说明：

- 哪些页面或路由会新增。
- 哪些已有页面会修改。
- 页面之间如何跳转。
- 哪些能力属于页面级逻辑，哪些属于可复用模块。

不要一开始就拆很多小组件。组件拆分应该服务于清晰边界、复用和可测试性，而不是为了显得工程化。

### 组件边界

组件设计应明确：

- 组件负责什么。
- 组件不负责什么。
- 输入 props 或外部依赖是什么。
- 输出事件或回调是什么。
- 组件内部是否持有状态。

好的组件边界应该让调用方简单，让复杂度集中在少数有价值的模块里。

不好的设计：组件只是把父组件传下来的字段原样展示，父组件仍然知道所有细节。

好的设计：组件封装一段完整交互，例如筛选条件编辑、批量操作确认、表格选择状态管理。

### 状态管理

区分不同类型的状态：

- **服务端状态**：来自 API 或后端数据源。
- **页面状态**：当前 tab、筛选条件、分页、排序。
- **表单状态**：输入值、校验结果、提交中状态。
- **交互状态**：弹窗开关、选中项、展开/收起。
- **全局状态**：跨页面共享的用户、权限、主题等。

不要把所有状态都放进全局状态。只有多个远离的页面或模块都需要读写时，才考虑全局状态。

### 数据流

说明数据如何进入页面、如何被转换、如何展示、如何提交。

至少覆盖：

- 数据从哪里来。
- 谁负责加载。
- 谁负责转换成视图模型。
- 用户操作后如何更新。
- 提交成功或失败后如何反馈。

避免让多个组件各自重复请求同一份数据，除非项目已有明确约定。

### 渲染与加载体验

前端设计要说明关键状态：

- 初始加载。
- 空状态。
- 部分加载。
- 加载失败。
- 提交中。
- 提交成功。
- 提交失败。
- 权限不足。

这些状态是用户体验的一部分，不是实现细节。

### 权限与可见性

明确权限对前端的影响：

- 用户是否能看到入口。
- 用户是否能看到数据。
- 用户是否能点击操作。
- 权限不足时展示隐藏、禁用还是提示。

不要只依赖前端权限判断；但前端设计需要明确用户看到什么。

### 错误与恢复

前端设计应说明用户遇到失败时如何继续。

例如：

- 请求失败是否可重试。
- 表单提交失败是否保留输入。
- 批量操作部分失败如何展示。
- 网络恢复后是否自动刷新。

错误文案和错误分类不在这里展开到完整错误码，但前端需要描述用户可感知行为。

### 可测试性

前端设计应让关键行为可验证。

优先描述外部行为，而不是内部实现：

- 用户看到什么。
- 用户点击什么。
- 系统如何响应。
- 哪些状态应该出现或消失。

不要把测试目标写成“调用了某个内部函数”。

## 工作方式

### 先尊重现有前端约定

设计前先观察项目已有结构：

- 框架和路由方式。
- 组件组织方式。
- 数据请求方式。
- 状态管理方式。
- 样式系统和组件库。

如果现有约定足够支持需求，优先沿用，不要引入新框架、新状态库或新目录结构。

### 先画边界，再列文件

前端设计应先说明页面、模块、状态和数据流的边界，再决定可能涉及哪些文件。

文件名和路径可以作为参考，但不要让设计退化成文件清单。

### 避免过度抽象

不要为了未来可能复用而提前抽象。

只有出现以下情况时才建议抽象：

- 同一交互在多个位置真实出现。
- 逻辑足够复杂，需要集中管理。
- 抽象能明显减少调用方需要理解的细节。
- 抽象能让测试更贴近用户行为。

### 保持前后端契约清晰

前端设计可以描述需要什么数据、什么时候提交、如何处理结果。

不要在前端设计里定义完整 API 细节；如果需要接口契约，应在输出中标记为待 API 设计确认。

## 补充检查点

本 reference 不规定输出结构；输出结构由 `design-change` 命令本体与 `.workflow/templates/changes/design/` 统一决定。

使用前端视角时，重点确认：

- 是否识别并沿用现有前端技术栈和目录约定。
- 页面、路由、模块和组件边界是否清楚。
- 状态管理是否区分服务端状态、页面状态、表单状态和交互状态。
- 数据流、权限可见性和用户可感知状态是否明确。
- 是否避免无必要的新框架、新状态库或新抽象。
- 是否标记需要 API 或产品进一步确认的问题。
