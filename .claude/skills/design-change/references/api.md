# API 设计

本文件用于指导 API 设计。当设计任务涉及接口定义、请求响应、协议、鉴权、兼容性时加载。

## 适用场景

- 需要设计前后端接口、服务间接口或外部开放接口。
- 需要定义请求参数、响应结构、状态码、错误格式。
- 需要设计鉴权、权限、幂等、分页、筛选、排序等接口行为。
- 需要修改已有 API，并判断兼容性影响。
- 需要对接第三方 API 或包装内部服务能力。

## 输入检查

进入 API 设计前，先确认以下输入是否可用：

- 产品设计中的能力、用户路径和可验证需求。
- 架构设计中的模块边界、服务边界和数据流。
- 现有 API 风格、命名、鉴权方式、错误格式和版本策略。
- 相关数据模型或领域概念。
- 调用方是谁：前端、移动端、内部服务、第三方开发者或 Agent。

如果缺少调用方、业务行为或数据概念，不要直接定义接口字段。

## API 设计关注点

### 围绕任务而不是表结构

API 应围绕调用方要完成的任务设计，而不是简单暴露数据库表或内部模块。

不好的设计：

```text
POST /user_status_updates
```

较好的设计：

```text
POST /users/{userId}/disable
```

接口名称应表达业务动作或资源语义，而不是泄漏内部实现。

### 调用方与使用场景

每个 API 都应说明主要调用方和使用场景。

至少明确：

- 谁调用。
- 在什么用户路径或系统流程中调用。
- 调用前需要什么状态。
- 调用后期望产生什么结果。
- 是否可能被批量调用、重复调用或异步调用。

### 资源、动作和边界

设计 REST 或 RPC 风格都可以，但同一项目内要保持一致。

REST 风格应关注：

- 资源名称是否稳定。
- HTTP 方法是否表达语义。
- 路径层级是否过深。
- 查询参数是否用于筛选、排序、分页。

RPC / action 风格应关注：

- 动作名称是否来自业务语言。
- 输入输出是否围绕完整任务。
- 是否避免把多个无关动作塞进一个接口。

### 请求设计

请求设计应明确：

- 路径参数。
- 查询参数。
- 请求体字段。
- 字段类型和约束。
- 必填/可选。
- 默认值。
- 示例。

不要只列字段名。重要字段要说明业务含义。

### 响应设计

响应设计应明确：

- 成功响应结构。
- 空结果如何表达。
- 是否返回完整资源、摘要资源或操作结果。
- 是否包含后续操作所需信息。
- 是否需要分页信息、游标、计数或链接。

响应应优先返回高信号信息，不要无差别返回所有内部字段。

### 错误设计

API 错误应可被调用方理解和处理。

至少定义：

- 错误码。
- 用户可读消息。
- 开发者可读原因。
- 是否可重试。
- 调用方下一步应该怎么做。

错误设计不要只写“返回 400”。需要说明为什么失败、调用方如何恢复。

### 鉴权与权限

说明 API 如何判断调用者身份和权限。

至少明确：

- 使用什么鉴权机制。
- 哪些角色或权限可以调用。
- 权限校验基于用户、租户、组织还是资源归属。
- 权限失败时返回什么错误。

### 幂等性

对创建、支付、批量操作、状态变更等接口，必须判断是否需要幂等。

如果需要幂等，说明：

- 幂等键来自哪里。
- 重复请求返回什么。
- 幂等记录保留多久。
- 部分成功时如何处理。

### 分页、筛选和排序

列表接口要明确：

- 分页方式：page/limit、offset/limit、cursor。
- 默认分页大小和最大分页大小。
- 可筛选字段。
- 可排序字段。
- 排序稳定性。
- 空列表响应格式。

### 兼容性和版本

修改已有 API 时，必须判断兼容性。

明确：

- 是否新增字段、修改字段、删除字段或改变语义。
- 是否会破坏已有调用方。
- 是否需要版本化。
- 是否需要过渡期、双写、兼容字段或弃用说明。

## 工作方式

### 先定义行为，再定义字段

不要一开始就写 JSON。

优先顺序：

1. 调用方要完成的任务。
2. 业务行为和状态变化。
3. 权限和失败场景。
4. 请求和响应结构。
5. 示例和兼容性说明。

### 示例必须真实

每个关键接口都应提供请求/响应示例。

示例中的字段值要贴近真实业务，不要使用 `foo`、`bar`、`string` 这类无意义占位。

### 不泄漏内部实现

不要把数据库字段、内部枚举、临时模块名直接暴露为 API 契约。

如果暴露内部字段会让后续演进困难，应设计稳定的外部语义。

### 明确失败路径

API 设计不能只写成功路径。

至少覆盖：

- 参数无效。
- 权限不足。
- 资源不存在。
- 状态不允许。
- 重复请求。
- 依赖服务失败。

## 补充检查点

本 reference 不规定输出结构；输出结构由 `design-change` 命令本体与 `.workflow/templates/changes/design/` 统一决定。

使用 API 视角时，重点确认：

- API 是否围绕调用方任务，而不是围绕表结构。
- 调用方、权限、幂等性和兼容性是否明确。
- 请求、响应和错误语义是否足以指导实现。
- 是否覆盖参数无效、权限不足、资源不存在、状态不允许、重复请求和依赖失败。
- 是否标记需要产品、数据或异常处理子域确认的问题。

## 质量检查

完成 API 设计后，检查：

- 是否先定义了行为，再定义字段。
- 是否明确调用方和使用场景。
- 是否覆盖成功和失败路径。
- 是否定义鉴权、权限和错误格式。
- 是否考虑幂等、分页、筛选、排序等通用行为。
- 是否判断兼容性影响。
- 是否提供真实示例。
- 是否避免泄漏内部实现细节。
