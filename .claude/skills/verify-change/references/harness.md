# verify-change harness 选择

验证前必须先构建或选择可重复的验证信号。没有可信验证信号，不能声称 change 通过。

## harness 优先级

按能覆盖真实风险的程度选择，不机械套用：

1. **单元测试**：适合纯函数、局部逻辑、明确边界。
2. **集成测试**：适合模块协作、数据库、队列、外部依赖适配。
3. **e2e 测试**：适合用户路径、跨层行为、UI/浏览器/完整业务流。
4. **HTTP / curl 脚本**：适合 API、服务端行为、鉴权、错误响应。
5. **CLI fixture + snapshot diff**：适合命令行、生成器、文本输出、文件变更。
6. **Headless browser 脚本**：适合 UI 状态、DOM、console、network 行为。
7. **Replay trace**：适合复现真实请求、事件流、日志或 payload。
8. **Throwaway harness**：适合难以通过正式测试触达的内部路径。
9. **Property / fuzz loop**：适合输入空间大、边界多、输出可判定的逻辑。
10. **Differential loop**：适合新旧版本、两种配置或两个实现的行为对比。
11. **HITL loop**：最后手段；如果必须人工操作，也要用脚本化步骤和结构化记录。

## harness 质量标准

验证信号应尽量满足：

- 快：可以在迭代中反复运行。
- 稳：结果确定，低 flaky。
- 准：失败信号指向具体承诺，不只是“没崩”。
- 可复现：他人可根据记录重新执行。
- 可追踪：输出能关联到 spec/design/tasks。

## 迭代 harness

如果现有 harness 太慢、太宽泛或不稳定，应先优化 harness：

- 缩小测试范围。
- 固定时间、随机数、环境和数据。
- 加强断言，让失败信号更尖锐。
- 捕获必要日志、截图、trace、录屏、自动化测试报告或交互日志。
- 将手动步骤转为脚本或结构化记录。

## 交互式 artifact 要求

涉及 UI、浏览器、CLI/TUI、终端式交互、实时流、可视化报表或其他用户可见行为时，harness 不能只证明命令退出成功，还应保留可审查 artifact。

常见 artifact 类型：

- 截图或录屏：适合 UI、可视化报表、交互状态和视觉回归。
- Trace 或自动化测试报告：适合浏览器、跨页面路径、网络请求和异步交互。
- 交互日志或 transcript：适合 CLI/TUI、终端式交互、实时流和 agent/tool 对话。
- 服务日志、浏览器 console 日志或事件流片段：适合失败路径、后台协作和实时更新。

如果无法采集 artifact，必须在 `verify.md` 中说明原因、影响范围和替代证据；不能静默把 UI/交互式验证写成“通过”。

`verify-change` 应记录选择了哪些 harness、为什么选择、覆盖哪些承诺、结果是什么。
