# describe-project 技能

## 目标文件

```text
docs/project.md
```

`docs/project.md` 是项目级长期上下文，作用类似项目的第二份 `CLAUDE.md`：它不指挥单次任务，而是让人和 Agent 快速理解这个项目是什么、怎么工作、哪些上下文必须优先知道。

## 必需内容

`docs/project.md` 必须覆盖以下内容：

1. **项目定位**：项目是什么、为什么存在、解决什么长期问题。
2. **用户与场景**：主要用户是谁、典型使用场景是什么。
3. **核心领域概念**：项目中反复出现、影响讨论和实现的术语、实体和边界。
4. **代码与文档结构**：稳定的顶层探索入口，用目录树加简短注释说明从哪里继续找信息；不维护文件级清单。
5. **技术与架构概览**：主要技术栈、系统边界、模块关系、运行时、数据边界和集成方式。
6. **容易犯错的边界**：开发中容易反复出错的长期工程边界，例如安全边界、状态归属、跨模块协作、运行依赖和验证方式。
7. **开发准则**：长期适用的开发、测试、验证和协作规则；能写成步骤的应写成步骤。
8. **重要文档列表**：后续工作应优先加载的文档入口，例如原型、规格、架构、接口、数据、运行手册和研究材料。

## 完整示例

下面是一份完整 `docs/project.md` 的示例形态，用来说明粒度和写法，不代表固定业务领域：

````md
# project

本文件记录项目长期 big picture。它帮助人和 Agent 在进入具体任务前，快速理解项目是什么、怎么工作、哪些上下文必须优先知道。

## 项目定位

本项目是一个面向中小客服团队的工单协作系统，用来把邮件、网页表单和聊天渠道的用户反馈统一成可分派、可追踪、可统计的工单。

## 用户与场景

- 客服主管每天查看待分派工单，调整优先级，并把复杂问题升级给合适负责人。
- 一线客服在工单详情中查看上下文、回复用户、记录处理结果，并在问题解决后关闭工单。
- 运营负责人每周查看处理时长、积压趋势和高频问题分类，用于改进服务流程。

## 核心领域概念

- **工单**：一次需要被跟进和解决的用户问题，是分派、处理、统计的基本单位。
- **会话**：围绕同一工单产生的消息流，可能来自邮件、网页表单或聊天渠道。
- **升级**：一线客服无法继续处理时，将工单转交给主管或专门处理人。
- **关闭**：用户问题已解决且无需继续跟进；关闭后的工单只保留查看和统计能力。

## 代码与文档结构

```text
support-desk/
├── frontend/        # 客服操作台和用户提交入口从这里开始找
├── backend/         # 鉴权、工单、消息、报表 API 从这里开始找
├── shared/          # 前后端共享的 DTO、状态枚举和错误码从这里开始找
└── docs/            # 产品规格、架构说明和运维手册从这里开始找
```

本节只提供探索入口，不维护源码文件清单；需要具体实现位置时，继续从对应目录命名、测试文件和长期文档搜索。

## 技术与架构概览

- 前端负责客服操作台体验和用户提交入口。
- 后端负责鉴权、工单读写、消息同步、权限校验和报表聚合。
- 数据库保存工单、用户、消息和处理记录；队列只处理邮件通知和报表生成等异步任务。
- 外部渠道接入只进入消息同步边界，不直接写客服操作台状态。

## 容易犯错的边界

- 不要在浏览器端信任租户 ID；跨租户数据访问必须由后端根据登录身份重新计算权限。
- 工单状态只能由后端状态机推进；前端展示状态不能反向决定工单是否可关闭或升级。
- 外部渠道消息可能重复投递；消息写入必须具备幂等键，不能只依赖接收时间去重。

## 开发准则

1. 新增跨端 API 时，先更新 shared DTO 和错误码。
2. 再补后端 service/route 测试，覆盖权限、状态流转和错误返回。
3. 然后接入前端 client 和页面状态，避免把服务端状态复制成长期本地状态。
4. 最后用端到端用例验证客服从列表进入详情并完成处理的主路径。

## 重要文档列表

- [工单生命周期规格](./specs/ticket-lifecycle.md) — 定义工单创建、分派、升级、关闭和重开规则。
- [客服操作台原型](./design/agent-console.md) — 展示客服处理工单的主要界面和交互顺序。
- [租户权限架构](./architecture/tenant-access.md) — 说明跨租户访问控制和后端授权边界。
- [部署排障手册](./runbooks/deploy.md) — 说明部署、回滚和常见运行问题排查步骤。
````

## 可选内容

在项目确实需要时，可以补充：

- 部署、运行或环境前提的高层说明。
- 重要外部系统、第三方服务或组织协作边界。
- 领域内容易混淆的概念对照。
- 已长期稳定的质量门禁或人工检查准则。

## 不应包含

`docs/project.md` 不应包含：

- 单次需求、任务状态、排期、当前进度或临时计划。
- 具体功能的验收条件、接口字段、UI 文案、测试用例或实现步骤。
- 过细的源码文件清单、函数名清单、类名清单或临时代码位置。
- 项目自身之外的维护流程说明。
- 未确认的猜测、临时调试结论或尚未稳定的方案。

## 写作要求

- 只写已确认或能从项目事实可靠推导的内容。
- 内容应稳定、可复用，适用于多个后续任务。
- 给足探索线索，不复制完整细节。
- 遇到过细内容时，应下沉到更具体的规格、设计、架构或运行手册文档。
- 遇到空章节时，优先删除；不要保留“待补充”噪音。
