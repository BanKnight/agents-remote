# clarify-intents 技能

## 定位

`clarify-intents` 是 workflow 的意图澄清会话：通过一次一个问题的追问，挖掘用户原始设计意图，并把已经稳定、已确认、尚未进入 roadmap 的意图维护到 `.workflow/intents.md`。

工作方式如下：

- relentlessly interview 用户，直到形成 shared understanding。
- 一次只问一个问题。
- 沿设计树逐个分支追问，先解决依赖关系。
- 如果问题可以通过阅读项目文件、长期文档或搜索资料回答，先自行查证，不把可查问题丢给用户。

`clarify-intents` 不是“一次记录一个意图”的工具。记录一批意图后，除非满足暂停或退出条件，否则继续追问下一个高价值问题。

## 主动触发

当用户提出模糊需求、粗糙想法、方向不清的问题，或表达“聊清楚 / 澄清一下 / 整理想法 / 帮我问清楚 / grill 一下”时，使用 `clarify-intents`。

不要在以下情况强行进入：

- 用户只是问概念或资料。
- 用户已经给出清晰实现指令。
- 用户明确要求直接设计、规划或实现。

## 输入契约

### 标准输入

每次执行都需要理解这些输入：

```text
用户当前表达的想法、问题或补充回答
.workflow/intents.md
.workflow/templates/intents.md
```

读取规则：

- `.workflow/intents.md` 是唯一写入目标，用于判断已有待讨论/待分配意图，避免重复、冲突或丢失上下文。
- `.workflow/templates/intents.md` 是写入格式来源；写入前必须读取并沿用模板字段与风格。
- 阅读 `.workflow/intents.md` 时，优先从文件后部往前读，先查看最近追加的待处理意图。
- 新增意图前，先在 `.workflow/intents.md` 中搜索关键词，确认是否已有相同、相近或冲突的意图。

### 条件输入

根据澄清问题的重要性与不确定性，按需加载这些输入：

1. **Project big picture**
   - 如果存在 `docs/project.md`，先读取它获得项目定位、用户场景、领域概念、长期准则和重要文档入口，再追问或记录意图。
   - 如果 `docs/project.md` 已列出与当前话题相关的 specs、design、architecture、runbooks 或 research，按需继续读取这些文档，避免重复询问已有结论。

2. **长期文档索引链路**
   - 如果判断某个背景信息很重要，但 `docs/project.md` 没有覆盖，按照 docs 索引规则层层查找：先读当前层 `index.md`，再进入相关子目录读取子目录 `index.md`，最后读取目标文档。
   - 只读取与当前澄清问题直接相关的文档；不要为了“全面了解”展开整棵 docs。

3. **运行态上下文**
   - 当需要判断某个意图是否已经进入 roadmap、是否和现有 change 冲突、或是否已经归档时，按需读取：

```text
.workflow/versions/index.md
.workflow/versions/<version>/changes/<change-id>/context.md
.workflow/versions/<version>/changes/<change-id>/progress.md
.workflow/archive/versions/
```

   - 这些文件只作为背景输入，不在 `clarify-intents` 中更新。

4. **项目文件与代码**
   - 如果用户的问题可以通过当前仓库文件、代码、测试或配置回答，先读取相关文件，再决定是否还需要追问。

5. **外部或专题资料**
   - 如果当前澄清依赖外部事实、技术文档、社区反馈或调研信息，按可用技能列表调用相应的研究、检索或文档技能。
   - 调用外部资料时，只提炼与当前意图判断相关的信息，不把调研过程写进 `.workflow/intents.md`。

## 输出契约

### 标准输出

每轮至少产生一个会话输出：

- 一个当前信息增益最高的问题。
- 问题前后的必要上下文，说明为什么现在问这个。
- 一个推荐答案或候选选项，方便用户确认、改写或反驳。

问题必须一次只问一个；不要在同一轮连续抛出多个问题。

### 条件输出

根据会话状态，按需产生这些输出：

1. **候选意图暂存说明**
   - 当用户回答暴露出可记录意图，但还未达到写入时机时，在会话中简短说明当前已稳定的候选意图。

2. **`.workflow/intents.md` 更新**
   - 当用户明确要求记录、即将暂停/退出、某条意图容易丢失上下文，或通常累计约 5 个稳定候选意图后，更新 `.workflow/intents.md`。
   - 还不清楚的意图写入“待讨论”。
   - 已经清楚但尚未进入 roadmap 的意图写入“待分配”。

3. **冲突或重复提示**
   - 如果发现相近、重复或冲突意图，优先向用户说明差异并请求确认；确认前不要直接追加重复条目。

4. **下一步衔接建议**
   - 当用户表示“进入 roadmap / 不用再问了 / 先到这里”，或主要分支已经澄清完毕时，简短说明已记录内容，并建议下一步使用 `plan-versions`。

## 不负责

`clarify-intents` 不负责：

- 规划 roadmap。
- 设计方案。
- 拆解任务。
- 执行实现。
- 做验证、长期沉淀或归档。
- 创建或更新长期 `docs/` 文档。

工作流层面的直接衔接只有 `plan-versions`：当意图池已经足够清楚并且用户希望进入版本/变更编排时，把下一步交给 `plan-versions`。

## 核心执行循环

每轮都按以下循环执行，直到满足暂停或退出条件：

1. 读取标准输入，并按条件输入规则补足当前轮必要背景。
2. 对用户当前表达中出现的每个新意图、候选意图或补充意图，逐条判断是否需要引入新的条件输入；如果 `docs/project.md`、docs 索引链路、运行态上下文、项目文件、代码或外部资料会影响意图理解、去重、拆分或记录，就先加载相应输入再继续。
3. 判断当前设计树中信息增益最高的一个问题。
4. 一次只问一个问题，并给出推荐答案，方便用户确认、改写或反驳。
5. 等待用户回答，不在同一轮连续抛出多个问题。
6. 用户回答后，先按第 2 步重新判断每个意图是否需要补充条件输入，再判断是否形成：
   - 新的候选原始意图；
   - 对已有候选原始意图的补充；
   - 需要继续澄清的模糊点；
   - 应留给后续 roadmap 编排或后续 change 阶段处理的问题。
7. 在会话中暂存已经稳定的候选意图，并尽量拆成后续可引用的小粒度条目。
8. 达到写入时机时，批量写入或更新 `.workflow/intents.md`。
9. 写入后继续追问下一个高价值分支，不因为单批意图已记录就结束。

## 追问策略

追问时优先沿设计树逐步覆盖这些方面，但不要一次全部询问：

- 用户原始目标：用户真正想改变什么，为什么现在要做。
- 使用者与场景：谁会用，在什么设备、环境和频率下用。
- 首期核心闭环：最小可用路径从哪里开始，到哪里算完成。
- 非目标：首期明确不做什么，避免 roadmap 被隐性需求污染。
- 领域术语：用户使用的词是否和 `docs/project.md` 或现有 docs 中的概念一致。
- Agent 生命周期：会话如何创建、运行、暂停、取消、结束、恢复。
- 控制能力边界：用户希望能执行哪些控制动作，哪些动作需要保护。
- 状态与输出：用户需要看到哪些状态、日志、结果、历史和错误信息。
- 多 Agent 统一语义：Claude、Codex 的差异哪些应暴露，哪些应被统一控制面隐藏。
- 响应式与 PWA：移动端、安装、离线、通知、后台恢复等预期是什么。
- 权限与安全：谁能访问、如何防误操作、是否涉及敏感输出或服务器权限。
- 并发与多会话：是否支持多个 agent、多个任务、多个用户或多设备同时操作。
- 失败与异常：断线、进程失败、agent 卡住、输出过大、服务器重启时用户期望什么。
- 可观测性与回溯：用户需要哪些记录来判断 agent 做过什么。

遇到模糊、冲突或过载术语时，应立即指出并给出更精确的候选表达。

## 意图拆解粒度

记录到 `.workflow/intents.md` 的每条意图会作为后续引用单元，因此应尽量满足：

- 单条意图聚焦一个可独立讨论、规划或实现的用户价值点。
- 单条意图尽可能小到后续可在约 15 分钟内完成一次实现尝试或验证。
- 如果用户表达的是一个大方向，应先在会话中拆成多个候选意图，再确认是否批量写入。
- 不要为了满足 15 分钟粒度而过度切碎语义；如果拆分会破坏用户意图完整性，可以保留较大条目，并在后续继续澄清。
- 不要把意图写成迷你设计文档；保留用户原始诉求，不提前塞入方案、任务或验收标准。

## 写入规则

当用户提出或确认原始意图，并达到写入时机时，更新：

```text
.workflow/intents.md
```

写入位置：

- 还不清楚的意图写入“待讨论”。
- 已经清楚但尚未进入 roadmap 的意图写入“待分配”。

写入内容只包括：

- 编号
- 原始意图

写入节奏：

- 不要每次用户回答后都立刻写入；优先在会话中维护候选意图。
- 通常每积累约 5 个稳定候选意图，再批量写入或更新一次 `.workflow/intents.md`。
- 如果用户明确要求记录、即将暂停/退出、或某条意图容易丢失上下文，可以少于 5 条也写入。
- 写入后继续追问下一个高价值分支，不因为单批意图已记录就结束。

写入要求：

- 只有用户确认后的内容才写入 `.workflow/intents.md`。
- 写入前先读取 `.workflow/templates/intents.md`，沿用模板字段和风格。
- 不更新 `.workflow/versions/index.md`。
- 不把 design、任务拆解、验收标准或实现方案写进 `intents.md`。
- 如果一轮回答同时暴露多个意图，先记录已经清楚的意图，再继续逐个追问剩余意图。

## 暂停与退出条件

`clarify-intents` 不应因为记录了一条意图就结束。

只有满足以下任一条件时，才可以暂停或退出：

- 用户明确表示“够了 / 暂停 / 先到这里 / 进入 roadmap / 不用再问了”。
- 当前设计树的主要分支已经覆盖，且没有新的模糊点或冲突点。
- 剩余问题已经明显属于后续 roadmap 编排或后续 change 阶段，而不是 intent clarify。

暂停或退出前，应简短说明：

- 已记录哪些意图。
- 哪些仍在“待讨论”或“待分配”。
- 推荐下一步使用 `plan-versions`。
