# 全栈状态同步设计原则

本文件记录 client-server 全栈系统中「状态同步」的**通用**设计原则，适用于任意全栈项目。本项目（agents-remote）会话重放领域的特有应用，见 [Claude2 进程模型与消息回放设计](./docs/design/message-replay.md)。

> ⚙️ 本文件仍在打磨中，作为独立文件维护以便迭代，不直接写进 CLAUDE.md 主体。

## 中心思想

客户端和服务端之间的关系，本质上是**如何合理化状态同步**。状态自身有多个正交的设计维度，这些维度**对称地**约束服务端和客户端两侧；**服务端作为数据源，必须先按这些规则保证自身状态维护正确**，客户端才有正确同步的基础。

贯穿其中的暗线是**「上下文充分性」**：全量同步之所以简单，正因为客户端始终握有全量上下文、能自行派生一切；按需同步之所以难，正因为客户端陷入**上下文不足**，无法再从局部数据自洽推出全局态——这是下文「两个根本问题」存在的根本原因，也是其中第二问「必须服务端配合」的由来。

## 状态的五个设计维度

这五条不是并列清单，而是**同一份状态对象的五个观察视角**，共同影响服务端与客户端的设计：

1. **生命周期**：常驻 vs 按条件消亡。要显式划定「何时生、何时灭」的边界，避免该消亡的不消亡（泄漏）、或该常驻的被误清。
2. **domain 划分**：按业务边界切模块，追求高内聚；切不开的横切关注点抽象成 shared/global。
3. **阶段**：初始化 / 更新 / 销毁。把状态的演进钉死在这三段里，问题会被压缩到「哪一段出错」，而不是散在整个生命周期。
4. **层次**：上下级 vs 平级。平级之间要互相隔离、由上层（管理层）协调——像组织架构，同层不直接串扰。
5. **同步**：全量同步 vs 按需同步。这是 client-server 关系的两种根本模式。

## 全量同步 vs 按需同步

| | 全量同步 | 按需同步 |
|---|---|---|
| 好处 | 实现简单；客户端自维护状态容易（始终有全量上下文） | 数据增长对用户体验影响微乎其微 |
| 坏处 | 数据增长后成为性能瓶颈 | 实现复杂；客户端长期上下文不足，需求一变就容易被迫改实现 |

> 我们当前正处在全量同步的「坏处」中（长会话回放变慢）。按需同步是要考虑的方向，但它引入「上下文不足」这个新难题，必须配套解决。

## 按需同步的两个根本问题

按需同步回到根本，是两个必须同时回答的问题：

### 问题 1：如何按需同步服务端状态——找切分边界

切分服务端状态的边界有两种思路：

- **固定参数划分**：分页 / 分大小等机械切分。
- **动态参数划分**：基于业务语义，例如时间范围、天数范围等条件。

### 问题 2：如何维护好自身状态——处理上下文不足

客户端先天上下文不足，关键点往往需要服务端配合：

1. **梳理好自身需要的状态**，和服务端协商如何获取缺失的部分。
2. **区分「按需同步服务端状态」时，对自己既有状态的影响**——哪些会被影响、哪些不受影响，要显式界定。
